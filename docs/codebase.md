# Codebase Guide

This document explains the architecture, modules, and key algorithms of the
`e2e_data_validation` project — a pipeline that converts Protocol Buffer 3 (PB3) binary
data to Apache Parquet and validates the round-trip fidelity field-by-field.

## 1. Architecture Overview

The pipeline has four stages executed in sequence from `notebooks/pipeline.ipynb`:

```
.proto file                    .pb3 binary
     │                              │
     ▼                              ▼
schema_inference.py          converter.py
     │                              │
     │ PyArrow Schema               │ PyArrow Table
     └──────────────────┬───────────┘
                        ▼
                  .parquet file
                        │
                        ▼
                  validator.py
                  (pb3 ⟷ parquet field-by-field)
                        │
                        ▼
                  ValidationResult
```

`generator.py` is used once up-front to create all the test datasets (`.proto`,
`.json`, `*_pb2.py`, `.pb3`).  The remaining three modules form the core pipeline.

## 2. Module: `generator.py`

**Class:** `DatasetGenerator`

**Purpose:** Create 8 self-contained test datasets that exercise every major Proto3
feature.  Each dataset is written to `datasets/<name>/` and contains:

| File | Description |
|---|---|
| `<name>.proto` | Schema definition |
| `<name>.json` | Human-readable test data |
| `<name>_pb2.py` | Compiled Python protobuf module (auto-generated by `protoc`) |
| `<name>.pb3` | Length-delimited binary serialization of the test records |

### How it works

1. `generate_all_datasets()` calls eight private `_generate_*` methods, one per
   dataset.
2. Each method builds a `proto_content` string (the `.proto` schema) and a `data` list
   (Python dicts representing records).
3. `_create_dataset(name, proto_content, data)` writes the `.proto`, serialises `data`
   to JSON, calls `protoc` via `subprocess` to produce `*_pb2.py`, then imports that
   module and serialises each record to the length-delimited `.pb3` binary format.

### Binary format written

```
┌─────────────────────────────────────────┐
│  varint(len(msg1_bytes))  │  msg1_bytes │
│  varint(len(msg2_bytes))  │  msg2_bytes │
│  ...                                    │
└─────────────────────────────────────────┘
```

Each message is preceded by a varint-encoded byte count (standard
[length-delimited protobuf stream](https://protobuf.dev/programming-guides/techniques/#streaming)).

## 3. Module: `schema_inference.py`

**Class:** `ProtoToParquetSchemaInference`  
**Exception:** `SchemaInferenceError`

**Purpose:** Parse a `.proto` file and produce a `pyarrow.Schema` that exactly mirrors
the proto message structure, preserving field widths and nullability without scanning
any data values.

### Public API

```python
schema = ProtoToParquetSchemaInference().infer_schema(proto_file: Path) -> pa.Schema | None
```

`self.errors` accumulates any non-fatal warnings.

### Type mapping table

| Proto3 type | PyArrow type |
|---|---|
| `int32`, `sint32`, `sfixed32` | `pa.int32()` |
| `int64`, `sint64`, `sfixed64` | `pa.int64()` |
| `uint32`, `fixed32` | `pa.uint32()` |
| `uint64`, `fixed64` | `pa.uint64()` |
| `float` | `pa.float32()` |
| `double` | `pa.float64()` |
| `bool` | `pa.bool_()` |
| `string` | `pa.string()` |
| `bytes` | `pa.binary()` |
| `enum` | `pa.int32()` |
| `message` | `pa.struct(...)` (recursive) |
| `repeated T` | `pa.list_(T)` |
| `map<K, V>` | `pa.map_(K, V)` |
| `optional T` | nullable `T` column |

### Key internal methods

#### `_parse_messages` — bracket-counting recursive parser

The naive regex approach breaks when messages are nested two or more levels deep.  This
method uses a **bracket-counting loop** to find the closing `}` at arbitrary depth:

```python
depth = 1
i = body_start
while i < len(content) and depth > 0:
    if content[i] == '{':
        depth += 1
    elif content[i] == '}':
        depth -= 1
    i += 1
message_body = content[body_start:i - 1]
```

After extracting the body it recurses into `_parse_messages(message_body,
f"{full_name}.")` to register nested messages under their qualified names (e.g.
`ComplexNested.Person`), then strips those nested blocks before parsing top-level
fields so inner fields don't pollute the outer schema.

#### `_strip_nested_message_blocks`

Removes nested `message { ... }` definitions from a message body using the same
bracket-counting technique.  Called immediately before field-level regex parsing to
prevent fields from inner messages appearing as spurious top-level columns.

#### `_find_main_message`

Returns the first message name that contains no `.` — i.e. the outermost, non-nested
message.  This is used as the schema root.

#### `_resolve_type`

Resolves a type name used inside a nested message by walking ancestor name prefixes.
For example, `Person` referenced inside `ComplexNested.Department` resolves to
`ComplexNested.Person` because the unqualified name lookup walks up through
`ComplexNested.Department` → `ComplexNested` → root.

## 4. Module: `converter.py`

**Class:** `PB3ToParquetConverter`

**Purpose:** Read a `.pb3` file, deserialise each message, convert to a `pyarrow.Table`
using the pre-inferred schema, and write a `.parquet` file.

### Public API

```python
converter = PB3ToParquetConverter(schema: pa.Schema)
success = converter.convert(pb3_file, parquet_file, proto_module_path, message_class_name)
```

### Internal pipeline

```
_read_pb3()            Read varint-delimited stream → list[protobuf message]
    │
    ▼
_message_to_dict()     Recursively convert each message to a Python dict
    │
    ▼
_records_to_table()    For each schema field, build a pa.Array from the dict list
    │
    ▼
pq.write_table()       Write the pa.Table to disk
```

### `_message_to_dict` — field type dispatch

```
field.message_type?
  ├─ yes + _is_repeated_field()  → [_message_to_dict(item) for item in value]
  ├─ yes + map_entry option      → _convert_map(value)
  └─ yes (singular)              → HasField() check → _message_to_dict or None
field is repeated scalar         → list(value)
field.type == TYPE_BYTES         → bytes(value)
field.type == TYPE_ENUM          → int(value)
else                             → value as-is
```

### `_is_repeated_field` — protobuf v7 workaround

The `protobuf 7.x` package uses a C extension backend (`_upb`) whose `FieldDescriptor`
instances do not expose `.label` as an instance attribute.  The method imports
`FieldDescriptor` from the class directly rather than reading it through an instance,
and falls back to checking `'Repeated' in type(value).__name__` if that also fails.
Map fields (which are internally `LABEL_REPEATED`) are explicitly excluded so they
continue to be routed to `_convert_map`.

### Array builders

| Method | Used for |
|---|---|
| `_create_struct_array` | `STRUCT` fields (nested messages) |
| `_create_list_array` | `LIST` fields (`repeated` scalars and messages) |
| `_create_map_array` | `MAP` fields (`map<K, V>`) |
| `pa.array(data, type=pa_type)` | All scalar primitives |

All four builders fall back to `pa.nulls(len(data), type)` on error rather than
crashing, so a single bad record does not abort the whole file.

## 5. Module: `validator.py`

**Class:** `DataValidator`  
**Return type:** `ValidationResult(success, message, details)`

**Purpose:** Read both the original `.pb3` file and the converted `.parquet` file,
normalise both into plain Python dicts, and compare them field-by-field.

### Public API

```python
result = DataValidator().validate(pb3_file, parquet_file, proto_module_path, message_class_name)
result.success   # bool
result.message   # summary string
result.details   # list of mismatch strings (up to 10)
```

### Why normalisation is necessary

When pandas reads a Parquet file back:

- `LIST` columns become **numpy arrays**, not Python lists — `isinstance(v, list)` fails and `pd.isna(numpy_array)` raises "truth value of array is ambiguous".
- `MAP` columns become **lists of `(key, value)` tuples**, not dicts.
- Scalar integer/float columns become **numpy scalar types** (`np.int32`, `np.float64`), not Python `int`/`float`.

`_normalize_parquet_record` / `_normalize_value` recursively convert all of these to
Python-native types before comparison.  `_is_na` wraps `pd.isna` in a try/except to
safely handle array-like values.

### `_compare_records` — recursive field comparison

```
for each key in pb_dict:
  pb_value is dict?     → recurse (handles STRUCT / nested messages)
  pb_value is list?     → compare element-by-element (handles LIST)
  pb_value is bytes?    → compare as bytes
  else                  → _values_equal(pb_value, pq_value) scalar compare

  parquet MAP comes back as list-of-tuples → dict() conversion before dict comparison
```

Mismatches are accumulated in a flat list with dot-notation paths
(`row 0.address.street`) to pinpoint exactly where values diverge.

## 6. Key Algorithms

### 6.1 Varint reading (used in both `converter.py` and `validator.py`)

Protobuf uses variable-length integer encoding for the length prefix of each message in
a stream.  Each byte contributes 7 bits of data; the high bit signals whether more bytes
follow.

```python
def _read_varint(f) -> int | None:
    result, shift = 0, 0
    while True:
        b = f.read(1)
        if not b:
            return None       # clean EOF
        byte = b[0]
        result |= (byte & 0x7F) << shift
        if not (byte & 0x80):
            return result     # high bit clear → last byte
        shift += 7
```

This is the **only correct way** to read a multi-message `.pb3` file.  Splitting on
`\n` is wrong and produces garbage.

### 6.2 Bracket-counting proto parser (in `schema_inference.py`)

See [section 3](#3-module-schema_inferencepy) above.  The key invariant: `depth`
starts at 1 (the opening `{` has already been consumed), increments on `{`, decrements
on `}`, and the loop exits when `depth == 0`.

### 6.3 Schema-from-proto (preserves exact integer widths)

By building the PyArrow schema directly from the `.proto` type declarations, the
pipeline guarantees that `int32` fields stay `int32` in Parquet — no upcasting to
`int64`.  If schema were inferred from data values (e.g. `pa.Schema.from_pandas(df)`),
PyArrow might conservatively widen small integer samples.

## 7. Data Flow Detail

```
generator.py
  └─ _create_dataset()
       ├─ write <name>.proto
       ├─ subprocess: protoc → <name>_pb2.py
       ├─ import <name>_pb2
       └─ write <name>.pb3  (varint-delimited binary stream)

pipeline.ipynb  (orchestration)
  │
  ├─ Step 1: DatasetGenerator.generate_all_datasets()
  │
  ├─ Step 2: ProtoToParquetSchemaInference.infer_schema(proto_file)
  │           └─ returns pa.Schema
  │
  ├─ Step 3: PB3ToParquetConverter(schema).convert(pb3, parquet, ...)
  │           └─ writes <name>.parquet
  │
  ├─ Step 4: DataValidator().validate(pb3, parquet, ...)
  │           └─ returns ValidationResult
  │
  └─ Step 5: Collect results → summary DataFrame → display
```

## 8. Known Limitations and Workarounds

| Limitation | Root cause | Workaround in this codebase |
|---|---|---|
| `protobuf 7.x` `FieldDescriptor.label` not an instance attribute | C extension (`_upb`) backend | Import `FieldDescriptor` from class; fall back to `type(value).__name__` inspection |
| Pandas returns numpy arrays for `LIST` columns | `to_pandas()` conversion | `_normalize_parquet_record` recursively converts to Python lists |
| Pandas returns list-of-tuples for `MAP` columns | `to_pandas()` conversion | `_compare_records` calls `dict()` on Parquet maps before comparison |
| `oneof` mutual exclusivity not preserved | No discriminated union in Parquet | Not fixed; documented in [compatibility analysis](Proto3_Parquet_Compatibility.md) |
| `enum` symbolic names lost | Stored as `int32` | Not fixed; documented in [compatibility analysis](Proto3_Parquet_Compatibility.md) |
| `google.protobuf.Any` / `Struct` not supported | No fixed schema possible | Not tested; documented as coverage gap in [compatibility analysis](Proto3_Parquet_Compatibility.md) |
